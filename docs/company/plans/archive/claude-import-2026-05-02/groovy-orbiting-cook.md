# Sprint-Cleanup-01 — Stop the "Stuck task recovered" Death Loop

## Context

Legion's recent sprint history shows a death loop. Of the last ~50 sprints in `project_id=3`:

- **31 FAILED**, **18 COMPLETED**, **1 ACTIVE**
- The most recent ~12 failures all share the same `last_error`: **`Stuck task recovered: Task ran for 2.1 minutes without completing`**
- Auto-Sprints (Learn-14 = Sprint 2891, etc.) and RCA-generated `Fix-49` → `Fix-58` sprints are all failing the same way
- The RCA daemon is dutifully creating new `Fix-NN: RCA — task stuck recovered` sprints to "fix" the killings — those sprints get killed too, fueling the loop

### Root Cause (4 colluding bugs)

1. **`backend/main.py:713`** — startup cleanup uses `max_running_minutes=1`. Any task that has been RUNNING for **60 seconds** when the backend restarts is forcibly recovered with the message above. Every other call site uses 30 minutes.
2. **`backend/app/services/sprint_lifecycle_graph.py:285-289`** — before the swarm even starts, the lifecycle graph mass-marks **all tasks** for the sprint as `RUNNING` with **identical microsecond `started_at` timestamps** (verified on Sprint 2891: 12 tasks all stamped `13:36:05.997085`-`.997109`). The swarm then processes them serially over many minutes; a restart at minute 2 wipes the entire sprint.
3. **Frequent backend restarts** (Docker rebuilds, supervisor recovery, manual restarts during recovery work) — each one triggers (1) and kills any sprint where (2) has happened.
4. **No execution semaphore + no per-project lock** in `sprint_lifecycle_graph` — the lifecycle daemon can race the agentic loop on the same project. The agentic loop already uses `get_project_lock(project_id)` (`agentic_loop_service.py:787-790`); the lifecycle daemon does not.
5. **RCA loop** — `root_cause_analysis.generate_fix_sprint` (`backend/app/services/root_cause_analysis.py:184`) signature dedup is signature-based and the messages vary just enough (`2.1`, `2.4`, `1.9` minutes) to escape dedup, so each restart spawns yet another `Fix-NN: RCA — task stuck recovered` sprint that itself becomes a victim.

### Verification of root cause

Direct DB evidence already collected:
- Task 9405 (Sprint 2891): `started_at = 13:36:05.997085`, `completed_at = 13:38:14.95452` → 2.1 min — exactly the threshold
- Sprint 2891 has 12 tasks with sequential microsecond starts (`.997085`-`.997109`) — proves mass-mark, not real execution
- Backend restarted at ~13:38:13, ~13:56:52 — startup cleanup ran, killed everything in flight
- `agent_swarm_service.supervisor_node` (lines 198-231) picks tasks ONE at a time from `task_batch[idx]` — confirming the mass-mark in lifecycle is unnecessary and harmful

### Intended Outcome

1. Stop the death loop **today** (one-time DB cleanup + a 1-character `main.py` fix).
2. Prevent the loop from re-emerging (per-task `RUNNING` transition + execution semaphore + per-project lock).
3. Stop RCA from manufacturing new victims (infra-only ignore-list).
4. Get sprint quality back to the post-Recovery-01 baseline (Sprint 2871 = 88.6, Exec 100, Learning 100).

## Plan (3 phases, ordered for safety)

### Phase 1 — Stop the bleeding (DB-only, no code, ~5 minutes)

Goal: clear the queue of doomed sprints and tasks so the system can come up clean.

#### A. One-time SQL cleanup (manual, single transaction)

Use the `legion-db` container. **Run inside one `BEGIN; ... COMMIT;` block so it's atomic and easy to roll back.**

```sql
-- 0. Sanity snapshot before cleanup
SELECT id, name, status, total_tasks, completed_tasks, failed_tasks
FROM sprints
WHERE project_id = 3
  AND (status = 'ACTIVE' OR (name LIKE 'Fix-%' AND status IN ('PLANNED','ACTIVE')))
ORDER BY id DESC;

BEGIN;

-- 1. Cancel currently-ACTIVE Sprint 2896 (5 PENDING tasks at risk)
--    fail_sprint() can't be called from SQL — use direct UPDATE then let
--    sprint_manager.recalculate_sprint_counts run on next lifecycle cycle.
UPDATE sprint_tasks
   SET status = 'CANCELLED',
       last_error = 'Sprint-Cleanup-01: cancelled to break stuck-task death loop',
       completed_at = NOW()
 WHERE sprint_id = 2896
   AND status IN ('PENDING','RUNNING');

UPDATE sprints
   SET status = 'CANCELLED',
       last_error = 'Sprint-Cleanup-01: cancelled to break stuck-task death loop',
       updated_at = NOW()
 WHERE id = 2896;

-- 2. Cancel any in-flight RCA Fix sprints generated by the loop
UPDATE sprint_tasks
   SET status = 'CANCELLED',
       last_error = 'Sprint-Cleanup-01: RCA loop victim',
       completed_at = NOW()
 WHERE sprint_id IN (
     SELECT id FROM sprints
      WHERE project_id = 3
        AND name LIKE 'Fix-%'
        AND (description LIKE '%task stuck recovered%'
             OR name ILIKE '%stuck%recover%')
        AND status IN ('PLANNED','ACTIVE')
 )
 AND status IN ('PENDING','RUNNING');

UPDATE sprints
   SET status = 'CANCELLED',
       last_error = 'Sprint-Cleanup-01: RCA loop victim',
       updated_at = NOW()
 WHERE project_id = 3
   AND name LIKE 'Fix-%'
   AND (description LIKE '%task stuck recovered%' OR name ILIKE '%stuck%recover%')
   AND status IN ('PLANNED','ACTIVE');

-- 3. Mark all stuck-recovered Auto-Sprints as CANCELLED instead of FAILED so
--    they stop polluting the failure-rate computation that drives the circuit
--    breaker. CANCELLED is correctly excluded from get_recent_failure_rate
--    (per Fix-42 learning).
UPDATE sprints
   SET status = 'CANCELLED',
       last_error = COALESCE(last_error,'') || ' [Sprint-Cleanup-01: reclassified from FAILED]',
       updated_at = NOW()
 WHERE project_id = 3
   AND status = 'FAILED'
   AND id IN (
       SELECT DISTINCT t.sprint_id
       FROM sprint_tasks t
       WHERE t.last_error LIKE 'Stuck task recovered: Task ran for%minutes without completing'
   );

-- 4. Verify counts before commit
SELECT status, count(*) FROM sprints WHERE project_id = 3 GROUP BY status;

COMMIT;
```

Acceptance: after this transaction, `SELECT count(*) FROM sprints WHERE project_id=3 AND status='ACTIVE'` returns `0`, and the recent failure-rate (`get_recent_failure_rate`) drops well below the 70% RCA circuit-breaker threshold.

### Phase 2 — Permanent code fixes (ordered low-risk → higher-risk)

All changes target Legion's own backend (project_id=3, repo `c:\code\Legion`).

#### B. Fix the 1-minute startup cleanup — `backend/main.py:713`

The single most impactful change. One character.

```python
# BEFORE (line 713)
recovered = await manager.auto_recover_stuck_tasks(max_running_minutes=1)

# AFTER
recovered = await manager.auto_recover_stuck_tasks(max_running_minutes=30)
```

**Why 30, not 60?** Matches the lifecycle daemon's existing default (`main.py:742`), matches the `auto_recover_stuck_tasks` function default, and is comfortably above the swarm's typical 5-15 min per-task ceiling but below the swarm's hard 60-min wait_for timeout (`agent_swarm_service.py:1180-1278`). A truly stuck task at minute 30 is also stuck at minute 60.

**Risk:** Negligible. The startup cleanup's job is to recover *truly* stranded tasks from a hard crash, not to interrupt healthy in-flight work.

#### C. Add execution semaphore + per-project lock to `sprint_lifecycle_graph.py`

This addresses the second cause: lifecycle daemon racing agentic loop on the same project.

**File:** `backend/app/services/sprint_lifecycle_graph.py`

**Module-level (top of file, near other imports):**

```python

import os

# Cap concurrent sprint executions across the entire process. Mirror the
# Ollama semaphore pattern in unified_llm_service.py:64.
LEGION_MAX_CONCURRENT_SPRINTS = int(os.getenv("LEGION_MAX_CONCURRENT_SPRINTS", "1"))
_sprint_execution_semaphore = asyncio.Semaphore(LEGION_MAX_CONCURRENT_SPRINTS)
```

**Inside the execute node, around the existing block at lines 285-298**, wrap the swarm call:

```python
from app.services.coordination import get_project_lock  # already imported elsewhere

async with _sprint_execution_semaphore:
    async with get_project_lock(project_id):
        # ... existing per-task RUNNING transition (see Component D below) ...
        # ... await execute_sprint_via_swarm(...) ...
```

**Reuse, don't reinvent:**
- `get_project_lock` already exists in `app.services.coordination` and is used by `agentic_loop_service.py:787-790` — same import path, same semantics.
- Pattern mirrors `_ollama_semaphore` in `unified_llm_service.py:64`.

**Default value of 1:** keeps current behaviour (one sprint at a time per process) but makes it tunable via env var without code change. Bumping to 2+ later requires verifying the swarm + Ollama queue can handle parallel sprints.

**Risk:** Low. Acquiring the lock is what the agentic loop already does. The semaphore default of 1 means no behavioural change for users running a single sprint at a time, but it formalizes the constraint.

#### D. Stop the mass-mark RUNNING race

The actual lie that drives the death loop: `sprint_lifecycle_graph.py:285-289` claims all 12 tasks are running at the *same instant* before the swarm has even started. Pick **D1**; fall back to **D2** only if D1 turns out to break a hidden caller.

##### D1 (preferred) — Per-task RUNNING transition inside the swarm

Move the `status = RUNNING` + `started_at = now()` write into `agent_swarm_service.supervisor_node` at the moment the task is actually picked up.

**File:** `backend/app/services/agent_swarm_service.py`

**At lines 198-231** (the existing "Pick the next task" branch in `supervisor_node`), after the `task = batch[idx]` line and before `return {...}`:

```python
# Mark THIS task RUNNING with a fresh started_at — moved out of
# sprint_lifecycle_graph to break the mass-mark race that caused the
# Sprint-Cleanup-01 death loop. The lifecycle graph used to stamp all
# tasks with identical microsecond timestamps before the swarm even ran,
# making them all look "stuck" to a 1-min cleanup at the next restart.
task_id_for_run = task.get("id")
if task_id_for_run:
    try:
        from app.core.database import AsyncSessionLocal
        from app.models.sprint_execution import SprintTaskDB
        from app.models.enums import TaskStatus
        from datetime import datetime, UTC
        async with AsyncSessionLocal() as run_db:
            db_task = await run_db.get(SprintTaskDB, task_id_for_run)
            if db_task and db_task.status == TaskStatus.PENDING:
                db_task.status = TaskStatus.RUNNING
                db_task.started_at = datetime.now(UTC).replace(tzinfo=None)
                await run_db.commit()
    except Exception as run_err:
        logger.warning(f"[Swarm] Could not mark task {task_id_for_run} RUNNING: {run_err}")
```

**And in `sprint_lifecycle_graph.py:285-289`, delete the mass-mark loop:**

```python
# DELETE these 4 lines:
# for t in tasks:
#     t.status = TaskStatus.RUNNING
#     t.started_at = datetime.now(UTC).replace(tzinfo=None)
# await db.commit()
```

The post-swarm orphan cleanup at `sprint_lifecycle_graph.py:317-329` already handles tasks that the swarm never reached (it queries for any RUNNING tasks left in the sprint and marks them failed). With per-task transition, that block now correctly catches *only* the task that was actually being processed when the swarm timed out — not the entire batch.

**Verification of feasibility:** Read of `agent_swarm_service.py:185-231` confirms `supervisor_node` is the *only* per-task pickup path (`current_task_index` increments only there). Each pickup is gated by `if not current_task or phase in ("idle", "complete_task")`. This is the single right place.

**Risk:** Medium. There is one subtle edge: if the supervisor is re-entered for a task that was already marked RUNNING (e.g., on retry), we don't want to overwrite `started_at`. The `if db_task.status == TaskStatus.PENDING` guard handles this — only PENDING→RUNNING transitions stamp a fresh time. Retries stay at the original `started_at`, which is what we want for stuck-task detection.

##### D2 (fallback) — Staggered timestamps in lifecycle graph

If D1 turns out to break some hidden invariant (e.g., a non-swarm execution path that depends on tasks being RUNNING at lifecycle time), apply this minimal patch instead at `sprint_lifecycle_graph.py:285-289`:

```python
from datetime import datetime, UTC, timedelta
now = datetime.now(UTC).replace(tzinfo=None)
# Stagger started_at across the swarm's serial pickup window so the
# 30-min stuck-task threshold is not tripped on tasks the swarm hasn't
# even reached yet.
for offset_idx, t in enumerate(tasks):
    t.status = TaskStatus.RUNNING
    t.started_at = now + timedelta(seconds=offset_idx * 5)  # 5s/task stagger
await db.commit()
```

This is strictly worse than D1 (it still lies about when each task started) but it's a 6-line change with zero risk to non-swarm callers. **Treat as escape hatch only.**

#### E. Add infra-only ignore-list to RCA — `root_cause_analysis.py`

Stop RCA from manufacturing Fix sprints for failure modes that are infrastructure issues, not code defects. The "stuck task recovered" pattern is the primary offender; the underlying cure is Components B+C+D, not a Fix sprint.

**File:** `backend/app/services/root_cause_analysis.py`

**At the top of `generate_fix_sprint` (around line 184)**, before any other logic:

```python
# Sprint-Cleanup-01: infrastructure-only failure modes that RCA must NOT
# generate Fix sprints for. These are caused by daemon/restart races, not
# by anything a code-fix sprint can address. Adding new patterns here is
# preferable to silencing RCA entirely.
_RCA_INFRA_ONLY_PATTERNS = (
    "stuck task recovered",
    "task ran for",
    "swarm did not process this task",
    "cleaned up on startup",
    "server restarted while running",
)

async def generate_fix_sprint(self, cluster: Dict[str, Any]) -> Optional[int]:
    """Auto-generate a Fix-NN sprint for an error cluster."""
    try:
        # Infra-only ignore-list — must be the FIRST check
        sample_lower = (cluster.get("sample_error") or "").lower()
        if any(pat in sample_lower for pat in _RCA_INFRA_ONLY_PATTERNS):
            logger.info(
                f"[RCA] Skipping Fix sprint for infra-only pattern "
                f"(sample: {sample_lower[:80]!r}). "
                f"These are addressed by Sprint-Cleanup-01 (B/C/D)."
            )
            return None
        # ... existing dedup + circuit breaker logic ...
```

**Why a list, not a single string:** future infra failure modes (Ollama queue full, MiniMax balance, etc.) can be added in one place without re-touching RCA control flow.

**Why ignore at `generate_fix_sprint` and not at cluster-detection time:** keeps the failure-cluster *visible* in metrics/dashboards (operators should still see infrastructure pain) but prevents the auto-sprint side effect.

**Risk:** Very low. RCA still runs; it just declines to create *one specific class* of sprint that we know is counterproductive.

## Risks & Open Questions

- **D1 retry edge case:** the `if db_task.status == TaskStatus.PENDING` guard prevents `started_at` from being reset on retry. Verify this is correct by reading the retry path in the swarm graph (LangGraph edge from `coder` → `supervisor` after a failed test) — if the swarm expects `started_at` to refresh on retry, switch to D2.
- **Semaphore default = 1:** matches today's effective behaviour. If users want to run two managed projects in parallel, they bump `LEGION_MAX_CONCURRENT_SPRINTS=2` and accept that Ollama queue depth doubles.
- **CANCELLED reclassification (Phase 1 step 3):** loses some failure history. We accept this; the failures were artifacts of the bug, not real signal. The `last_error` tag preserves the audit trail.
- **RCA ignore-list coverage:** the 5 patterns cover the loop we know about. New infra failure modes will need to be added; this is by design (visible, intentional, one-line change).
