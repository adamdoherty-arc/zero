# Why does the LLM Console say "0 calls" and do sprints look duplicated?

## Context — what's actually happening

### 1. "Showing every LLM call produced by sprint task #57 (0 calls found)"
The URL is deep-linked with `?sprint_task_id=57`. When that query param is present, [LLMConsole.tsx:917-924](frontend/src/pages/LLMConsole.tsx#L917-L924) **replaces** the live `activeCalls + recentCalls` stream with a historic query against `GET /llm-console/calls?sprint_task_id=57&limit=200`.

- Task #57 = sprint_id 12, title "Upgrade fastmcp 2.14.7 → 3.2.4", status COMPLETED
- DB check: `SELECT COUNT(*) FROM llm_call_details WHERE sprint_task_id=57` → **0 rows**
- This is *correct*. That task's work went through a code-only path (no LLM call was made, or the call wasn't tagged with `sprint_task_id=57`).
- The 155/141/14 counters at top are **not** filtered — they're the global 24h stats, which is why they seem to contradict the empty list.

**The confusion** is that the banner at top says "0 calls found" *and* the list below says "No LLM calls yet — Calls will appear here in real-time as they happen." Two messages, same empty state, different meanings. Click **Clear filter** and 155 calls return.

### 2. "No provider overrides"
Expected. FE-Transparency-01 retired the Learn-18/19/20 override rows on 2026-04-22 (memory note `vllm_transition_2026_04_22.md`). DB confirms: `SELECT COUNT(*) FROM prompt_templates WHERE provider_override IS NOT NULL` → 0. `active_provider=vllm` is canonical now; absent override rows are *not* a regression.

### 3. "Several completed sprints look duplicated"
No DB duplicates — 9 unique sprints for project_id=1 with unique IDs. But **6 of them share the visually-identical prefix** "Dependency review for Legion":

```
18: Deps-07: Dependency review for Legion [completed]
16: Deps-06: Dependency review for Legion [completed]
14: Deps-05: Dependency review for Legion [completed]
12: Deps-04: Dependency review for Legion [completed]
 9: Deps-03: Dependency review for Legion [completed]
 6: Deps-02: Dependency review for Legion [completed]
 4: FE-Transparency-01: vLLM-only routing + sprint system honesty
 3: Sprint 0: Product Documentation [cancelled]
 1: Deps-01: Dependency review for Legion [completed]
```

Each is a separate daily dependency review run (generated by [dependency_review_service.py:361](backend/app/services/dependency_review_service.py#L361) which uses `f"Deps-{sprint_num:02d}: Dependency review for {project_name}"`). They read as duplicates because the only distinguishing token is the two-digit number at the very start.

### 4. Bonus — "Disconnected — Backend unreachable" banner
Backend is actually healthy (`/health`, `/api/llm-console/stats`, `/api/llm-console/calls` all 200). The banner comes from [Sidebar.tsx:300](frontend/src/components/Sidebar.tsx#L300) and fires on WebSocket state, not REST. `useWebSocket.ts` gives up after 3 reconnect attempts on `ws://localhost:8005/ws/events` and stays Disconnected until the page is reloaded. Not part of this plan to fix — separate issue.

## Recommended change — clarify the filtered empty state

Single small frontend fix in [LLMConsole.tsx](frontend/src/pages/LLMConsole.tsx) so the next time someone deep-links into an LLM-silent task, they aren't left wondering whether the page is broken.

### Files to modify
- `frontend/src/pages/LLMConsole.tsx` — around line 938-941 (where `allCalls` is memoised) and the empty-state render block below the LLM Calls list (currently renders the generic "No LLM calls yet").

### What to change
1. When `sprintTaskIdValid && filteredCalls.length === 0`, swap the generic `<EmptyState icon="..." title="No LLM calls yet" ... />` for a filter-aware copy:

   > **This sprint task produced 0 LLM calls.**
   > Task #57 may have completed via a non-LLM code path, or its calls were logged without the `sprint_task_id` tag. [Clear filter] to view all recent calls.

   The `[Clear filter]` text should be a button wired to the existing `handleClearTaskFilter` handler at line 930.

2. Also consider moving the top banner copy from "(0 calls found)" to count the filter hits more explicitly: "(filtered — 0 of 155 calls matched)" so the relationship to the stats counters above is obvious.

### Optional nice-to-have (NOT required)
- Date-stamp Deps sprint names in [dependency_review_service.py:361](backend/app/services/dependency_review_service.py#L361): `f"Deps-{sprint_num:02d} ({date}): Dependency review for {project_name}"` so the 6 runs are visually distinct. Skipping this in scope — it's a separate concern and would churn historical sprint names if done retroactively.
