## What's implemented (as of 2026-04-22)

From `backend/app/services/reachy_presence_service.py`:
- `start_meeting_mode(meeting_id)` → returns state, plays `welcoming1`, spawns `_meeting_loop()`.
- `_meeting_loop()` polls `reachy_service.get_doa()` every 3 s; when `speech_detected` is true and the angle changed by > 0.15 rad, calls `look_at(x=cos(angle), y=sin(angle), z=0, duration=0.6)`.
- Every 45 s, regardless of DoA, plays `attentive1` so the robot looks engaged even in silence.
- `stop_meeting_mode()` cancels the loop, plays `understanding2`, clears state.
- Pomodoro tick, hourly chime, and presence beat all short-circuit when `_meeting_mode` is True to avoid gesture collisions.

Wired into the existing meeting pipeline:
- `meeting_recording_service.start_recording()` calls `start_meeting_mode(meeting.id)`.
- `meeting_recording_service.stop_recording()` calls `stop_meeting_mode()`.
- Manual control exposed at `/api/reachy/presence/meeting/start|stop|status`.
- UI button in `ReachyMotionLibraryPage` → Modes panel (commit `8d28e24`).

**Live-verified 2026-04-22**: meeting mode started, ran 17 s, polled DoA three times with real angles (1.45 → 1.34 → 1.92 rad), stopped cleanly.

## Known gaps (ordered by impact)

### Gap 1 — No speaker diarization (biggest coverage hit)
DoA gives a *direction* but not an *identity*. Reachy looks toward whoever is currently speaking but doesn't know who that is, which means:
- Can't address meeting participants by name
- Can't surface "Adam talked 70% of the meeting" in the summary
- Can't bias attention toward a specific speaker ("track Pierre whenever he speaks")

`DailyMeetings` on the host already runs `pyannote` for diarization in its transcription pipeline — but that output is never fed back to Reachy. The fix is a subscription model: Reachy reads the live diarized transcript, maps diarized speaker → last known DoA angle, and remembers identity across silence gaps.

### Gap 2 — No nod-on-highlight
The meeting transcript is streaming in real time, but Reachy doesn't *react* to it. If you say "we agreed on Thursday" there's no `understanding1` gesture. If someone says something surprising, no `surprised1`. The robot looks attentive but not comprehending.

The plumbing exists: `meeting_ws.py` has a WebSocket that streams transcript chunks. A new `meeting_highlight_service.py` could subscribe, run a lightweight classifier (keyword list + LLM tag every ~30 s), and dispatch emotions on match.

### Gap 3 — No persona-aware meeting modes
Meeting mode runs whatever persona was active before the meeting started. So if `cosmic_kitchen` was on and you walk into a board meeting, the robot is still in sitcom mode. Integration idea II-002 (in `INTEGRATION_IDEAS.md`) proposes snapshotting the persona on `start_meeting_mode`, switching to a meeting-appropriate one (e.g., `victorian_butler` for formal), and restoring on stop.

Harder: *which* meeting-appropriate persona depends on the meeting itself. Use the title keyword or the calendar event's attendee list to pick:
- 1:1 → `noir_detective` (terse, probing)
- board / formal → `victorian_butler`
- standup → `hype_bot`
- brainstorm → `cosmic_kitchen` (the default)
- default → `victorian_butler`

### Gap 4 — No Reachy-mic audio capture integration
Zero currently records meetings via the host's primary USB mic through `DailyMeetings`. The Reachy itself has a spatial mic + DoA. If the meeting recording used the Reachy's mic, DoA would be **synchronized with the transcript** — you'd know which angle produced which sentence without any correlation logic.

Requires routing the Reachy mic into the `meeting_recording_service` pipeline. Daemon exposes the audio via WebRTC :8443; we'd need a subscriber on the host that pipes it into the same `meeting_audio_capture` ring buffer.
