# Reachy Audit - 2026-05-05

## Trigger

Safety-first re-review after the physical robot shook uncontrollably during assistant activation.

## Main finding

The companion policy correctly defaulted to `body_motion_enabled=false`, but the lock was not global. Legacy routes and background services could still enable daemon/watchdog/motors or dispatch motion. This made the new companion UI safer than the underlying system, which is not acceptable for hardware.

## Fixes landed

- Added `backend/app/services/reachy_motion_policy.py`.
- Locked `/api/reachy/assistant/activate` to voice-only defaults.
- Guarded direct motion routes, daemon start/restart/watchdog enable, motor enable, realtime motion tools, realtime wobbler, classic voice gestures, HA gestures, presence modes, sequence playback, move recorder/replay, and radio.
- Changed recorded-move stop to disable motors instead of re-enabling.
- Migrated `meeting_vector_service.py` off direct Ollama embeddings and onto the shared client.
- Moved carousel Stage-8 VLM default from hard-pinned `gemini-2.5-flash-lite` to `ZERO_GEMINI_VISION_MODEL` / `ZERO_VLM_MODEL` / `gemini-flash-latest`.
- Recovered host-agent reachability by moving the supervisor surface to `:18796` because Docker Desktop held `:18794` in a bound state, and by running `supervisor_only:app` for daemon/status safety while the full audio/camera host agent is unstable.
